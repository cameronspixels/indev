Learning Wordpress

A traumatic experience |

Wordpress

09/03/2017

by Cameron Barnes

Having attended WordCamp Manchester 2017, I was opened up to the possibilities and relative ease of WordPress, something became all the more apparent when looking towards job opportunities. The amount of jobs where understanding of WordPress was a requirement, even for the front-end developer, was seemingly endless. Looking for a job soon, this seemed like a good time to get learning.

Setting Up

The first port of call is of course the official website. Handily, there’s a button to take you to download WordPress from the off. To set up, WordPress of course requires a database, and typically, a server to be run on.

While the codex is a great resource, when looking to really get off the ground, I came across Tania Rascia’s website. She’s got a helpful written tutorial on setting up WordPress locally and getting started with theming. Traversy Media's YouTube tutorial on creating a WordPress theme was another resources which affirmed learning and was of great use.

The easiest way to do this is creating a server run locally on your machine. For the purposes of this project, I used MAMP run on a MacBook Pro. This program is created to run on MacOS (M) and enable locally Apache (A), MySQL (M) and PHP, Perl or Python (P). There, MAMP.

Yes, I used Wikipedia.

There’s likely little setup required – if you don’t run local servers, MAMP’s default port of 8080 will work without issue, and you’ll be able to access it immediately. All you need to do is locate and extract the WordPress files into the /htdocs folder, and you’re up and running with the installation.

Things to Learn

Having used PHP to a reasonable degree in the past, there wasn’t the greatest learning curve when creating WordPress theme. I had an understanding of the code required for includes and the like, as well as a grasp of how they’d compile in HTML.

What did need learning was WordPress’ structure – so to say, the exact .php files required for a theme and how they interact. For instance, you have your standard index.php, header.php and footer .php, and page.php – as well as the style.css, which also defines the theme’s details.

On top of that, you need a functions.php, which essentially overlooks everything and defines specific actions in the .php. A front-page.php file details a static homepage if one is defined in the customising settings, and is read separately from index.php or page.php. The designated posts page, once using a static homepage, then needs to be created with home.php.

The Project

What I aimed with this was re-create my portfolio website in WordPress to get to know the platform. I wasn’t sure whether or not I would use the WordPress version as my website going forward, but in the best interest of learning to theme WordPress, it would at least be functional.

You might notice that this website isn’t the WordPress version. I chose not to use it, but if you head over to www.cameronspixels.co.uk/wordpress, you can see what I managed to get up to.

‘Why?’ Moments

I’d just come to the point of changing the navigation in the header.php file, from static HTML to one that was dynamically from the site structure. This was simple using the codex as reference.

Immediately, this was un-styled. Given that I tend to assign classes to the majority all my HTML elements and target classes in the CSS, the WordPress navigation (with no classes) didn’t inherit my styling.

Admittedly a small issue, this was fixed by altering my CSS classes to target semantic elements. I imagine that there would be a workaround through something like the functions.php file – a way to instruct WordPress to assign classes to dynamically generated elements. The face of the issue, however, is frustrating when inheriting a stylesheet not written with WordPress in mind.

Static or Dynamic?

Similar migration problems occurred when debating which content was to be made dynamic, and which would remain static.

I’d initially attempted to make all of the homepage content dynamic by using posts, and fetching them using PHP. Once more, I ran into issues with design. My original website had sections which were alternating between black and white, breaking up the page stylistically. While I could style posts semantically so that they would adhere to the grid positioning, I could find no simple way to differentiate between which should have a black background and which should remain white.

I had a similar issue with my headers, in that each page had a different colour as the background. My solution to this, perhaps convoluted, was to create and import separate .php files for each header. And if loop in the page.php would decide which header to import, determined by the page url.

Plugins

One of the main things that I wanted to evaluate here was whether or not WordPress plugins eased development. For this, I chose YoastSEO, the most popular search engine optimisation plugin on WordPress. I’d learnt a reasonable amount about SEO recently, so was interested to see how Yoast handled it.

Having gone through the configuration process, which consisted largely of me providing the name of the site and any social media links, I noted that descriptions and keywords for most pages were not generated. Even titles needed filling in. I was still completing a lot of the SEO process myself, just in the form of a graphic interface as opposed to the text editor.

The Product

On the whole, I’d conclude that use of WordPress (and a CMS in general) is largely situational. There are a number of reasons that I’m likely opting to remain with the static version of my own site, which I’d consider when deciding if WordPress is appropriate in the future.

This project gave me an understanding of dynamic content, how WordPress integrates that static content with static pages, and an overview of potential plugin benefits.

However, while the WordPress interface simplifies websites, for someone like myself who is already tuned to the more technical site, it’s quicker and easier to dive into the code when adding or altering content. Themes felt like a longer route, and plugins like Yoast then came across as little more than another interface for SEO.

Though this project may not have been the best to learn initially, it did give me a grasp of custom themes and plugin usage. Understanding that context, I’m open to learning more about WordPress. With a better understanding of PHP and other technologies involved, I’m interested in trying out different options in different circumstances.
